# פורטל גביש - מה קרן רש"י צריכים לעשות בשלב הפרודקשן

> מסמך חי. מתעדכן בכל פעם שמצטבר מידע חדש. עודכן לאחרונה: 27.05.2026

## מטרת המסמך

הפורטל (אתר קהילת גביש) נבנה כפיילוט וחי כרגע בחשבונות הפרטיים של רועי (Supabase + Vercel + Resend). המסמך מרכז את כל מה שצריך לעשות כדי להעביר אותו לסביבת הפרודקשן של קרן רש"י, כך ששום דבר לא ייפול בין הכיסאות. חלק מהסעיפים דורשים החלטה של רש"י (בחירת דומיין, כתובת שולח, רשימת מנהלים).

---

## 1. Supabase (בסיס נתונים, התחברות, אחסון)

- [ ] **העברת הפרויקט** מהארגון של RT-AI לארגון של רש"י. אפשר דרך Supabase Settings ← Transfer Project (שומר נתונים + אחסון), או ייצוא וייבוא לפרויקט חדש בארגון של רש"י. Project ref נוכחי: `ugnxgqwzpkkwsktppuus`, אזור `eu-central-1` (פרנקפורט, הכי קרוב לישראל - לשמור אותו אזור).
- [ ] **SMTP מותאם** - כרגע משתמש בברירת המחדל של Supabase (שולח `noreply@mail.app.supabase.io`, מוגבל ל-4 מיילים בשעה). להגדיר Resend על דומיין של רש"י, כך שהשולח יהיה `פורטל גביש <noreply@gavish.{דומיין-רש"י}>`.
- [ ] **מדיניות סיסמאות חזקה** - Supabase Auth ← Password Settings ← לדרוש לפחות 10 תווים + שילוב אותיות/מספרים/סימנים.
- [ ] **תבניות מייל בעברית** - לוודא שתבניות ה-3 אירועים (אישור הרשמה, magic link, איפוס סיסמה) הועברו. משתמשות בתבנית `{{ .SiteURL }}/auth/confirm?token_hash={{ .TokenHash }}&type=...` (לא `{{ .ConfirmationURL }}` - זה שובר את הזרימה).
- [ ] **כתובות Auth** - להסיר `http://localhost:3000/**` מ-uri_allow_list, להגדיר site_url לכתובת הפרודקשן הסופית.
- [ ] **2FA** מופעל על חשבון ה-Supabase של רש"י.
- [ ] **מנהלים אמיתיים** - לקדם דרך SQL את המנהלים האמיתיים (רשימה תתקבל מרועי), להוריד הרשאות אדמין מחשבונות בדיקה.
- [ ] **ניקוי נתוני בדיקה** - למחוק את חשבונות הבדיקה ואת הפוסטים לדוגמה.

## 2. Vercel (אירוח האתר)

- [ ] להעביר את הפרויקט ל-Team של רש"י (או להמשיך לפרוס מהריפו של רועי עד שרש"י מוכנים).
- [ ] לעדכן משתני סביבה: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `NEXT_PUBLIC_SITE_URL` - שיצביעו על הפרויקט החדש.
- [ ] לעדכן CSP ב-`vercel.json` (connect-src + img-src) שיאפשר את דומיין ה-Supabase החדש.
- [ ] להוסיף דומיין מותאם ב-Vercel ולכוון אליו DNS.

## 3. דומיין ו-DNS

- [ ] **SPF** - `v=spf1 include:_spf.google.com ~all` (להתאים לספק המייל בפועל).
- [ ] **DKIM** - להפעיל אצל ספק המייל ולהוסיף את רשומת ה-TXT שנוצרת.
- [ ] **DMARC** - להתחיל ב-`p=quarantine`, לשדרג ל-`p=reject` אחרי 48 שעות נקיות.
- [ ] **HSTS preload** - להגיש אחרי בדיקת תת-דומיינים.
- [ ] `/.well-known/security.txt` - כבר מחווט, רק לוודא ש-`Contact:` מצביע לכתובת של רש"י.

## 4. מייל ההרשמה לקהילה (טופס /join) - מגיע לנועה

טופס ההרשמה באתר שולח מייל אל **noana@rashi.org.il** בכל הרשמה, דרך Resend. כרגע (בפיילוט) המייל מגיע רק לרועי, כי הדומיין עוד לא מאומת ב-Resend. כדי שזה יעבוד לנועה בפרודקשן עם מיתוג גביש נקי:

- [ ] **לאמת את rashi.org.il ב-Resend** - resend.com/domains ← Add Domain ← `rashi.org.il`.
- [ ] **להוסיף את רשומות ה-DNS** ש-Resend מייצר (~3-4 רשומות): DKIM TXT (`resend._domainkey`), SPF TXT + MX על תת-דומיין `send.rashi.org.il`, ו-DMARC מומלץ.
  - **הערכים המדויקים נוצרים בזמן הרישום ב-Resend - יש למשוך אותם מהדאשבורד באותו רגע. אי אפשר לכתוב אותם מראש.**
- [ ] **חשוב להרגעת ה-IT:** הרשומות מבודדות לתת-דומיין `send.` + סלקטור DKIM נפרד, ולכן **אינן נוגעות במייל הקיים של Google Workspace** - noana@ ו-gavish@rashi.org.il ממשיכים לעבוד כרגיל.
- [ ] **אחרי שהאימות עובר** - להגדיר `RESEND_FROM` ל-`קהילת גביש <gavish@rashi.org.il>` (או כתובת @rashi.org.il אחרת) ולהוסיף `RESEND_API_KEY` ל-Vercel (Production/Preview/Development), ולפרוס מחדש.
- [ ] תוצאה: הרשמות אמיתיות נוחתות אצל נועה עם המיתוג הנקי "קהילת גביש", ללא הפניה ל-rt-ai, וללא נפילה לספאם.

**מצב נוכחי בפיילוט:** המפתח ב-Resend הוא "שליחה בלבד" (לא יכול לרשום דומיין). השולח כרגע `onboarding@resend.dev` ומגיע רק לבעל החשבון (רועי). המייל אינו ניתן לתשובה (אין reply-to), כתובת הנרשם מופיעה בגוף המייל. אם השליחה נכשלת, הטופס מציג הודעה חיננית "פנו ישירות אל noana@rashi.org.il".
